# 0319 与 0323 两批 Task2 实验详细解析

## 1. 这份文档在解析什么

这份文档面向当前仓库里两批和 `Task2: benign(良性肿瘤) vs no_tumor(非肿瘤性息肉)` 相关的核心实验：

- `0319/`：基线实验 + 第一轮 recipe-tuning + `ConvNeXt img320` 多 seed 验证
- `0323/`：基于 `0322dataset` 的 JSON 标注增强实验，包括 ROI 裁剪、4 通道 mask、强增强、Mixup/Focal、阈值优化、多 seed、集成、XGBoost stacking

本文不是简单复述原始总结，而是把以下几件事串起来讲清楚：

1. 每批实验到底在回答什么问题
2. 各组实验之间的逻辑关系是什么
3. 为什么某些实验会涨点，为什么某些实验看起来“很高级”却没有继续涨
4. 现有结果能支持哪些结论，不能支持哪些结论

---

## 2. 最重要的前提：`0319` 和 `0323` 不是同一套数据

这件事必须先讲清楚，否则后面所有“谁最好”的讨论都会混乱。

### 2.1 `0319` 实验使用的数据

`0319` 目录里的训练脚本直接指向：

- `../0316dataset/Processed/task_2_train.xlsx`
- `../0316dataset/Processed/task_2_test.xlsx`

对应脚本示例：

- `0319/20260319_task2_ConvNeXtTiny_adamw_warmupcosine_classweight_img320_3.py`
- `0319/20260319_task2_SwinTiny_adamw_warmupcosine_classweight_2.py`

而 `0319/实验报告总结.md` 给出的数据规模是：

- train = `1303`
- test = `564`
- train 类别分布：`benign=309`, `no_tumor=994`
- test 类别分布：`benign=129`, `no_tumor=435`

### 2.2 `0323` 实验使用的数据

`0323/scripts/` 里的实验脚本直接指向：

- `0322dataset/task_2_train.xlsx`
- `0322dataset/task_2_test.xlsx`

例如：

- `0323/scripts/20260323_task2_SwinV2Tiny_full4ch_strongaug_7.py`
- `0323/scripts/20260323_task2_SwinV2Tiny_balanced_mixup_9.py`
- `0323/scripts/20260323_task2_multiseed_train.py`

`0323/docs/0322_实验总结.md` 给出的数据规模是：

- train = `1229`
- test = `523`
- train 类别分布：`benign=309`, `no_tumor=920`
- test 类别分布：`benign=129`, `no_tumor=394`
- 一个额外的重要条件：`0% 患者重叠`

### 2.3 这意味着什么

这意味着：

- `0319` 的 `0.6372` 和 `0323` 的 `0.6501` 不是严格同场比较
- 它们训练样本数不同，测试样本数不同，类别分布也略有差异
- `0323` 还明确强调了“训练集与测试集 0% 患者重叠”，泛化条件更苛刻

因此，正确的阅读方式不是把两批实验混成一张排行榜，而是：

- 先看 `0319` 解决了什么问题
- 再看 `0323` 又新增了什么信息和什么新瓶颈

## 3. 任务本身为什么难

从 `README_0322dataset.md` 和 `0323/docs/0322_实验总结.md` 可以归纳出三个决定性事实。

### 3.1 类别不平衡很明显

无论是 `0319` 还是 `0323`，`Task2` 都大致是 `benign:no_tumor ≈ 1:3`。

这直接带来两个现象：

- 模型很容易学成“偏向 `no_tumor`”
- 单看 `Acc` 很容易被误导

这也是为什么两个批次的文档都把重点放在：

- `macro-F1`
- `benign recall`
- `benign f1`

而不是只看 accuracy。

### 3.2 两类样本都“有病灶”

`0322dataset` 的说明里明确写了：

- `benign` 和 `no_tumor` 两类都有 `gallbladder polyp`、`pred` 等病灶标注
- 模型学的不是“有没有病灶”，而是“病灶是什么性质”

这点非常关键。很多医学二分类任务之所以容易，是因为正负类在视觉上有显著结构差异；但这里不是。

### 3.3 真正区分两类的，可能更接近病理属性而不是稳定视觉模式

`0323/docs/0322_实验总结.md` 的关键洞察是：

- 两个类别本质上更像“有 polyp 的胆囊” vs “有 polyp 的胆囊”
- 差别在于 polyp 的病理性质，而不是稳定、显著、可视化的影像形态

其中 `adenoma` 标注虽然有区分力：

- benign 中比例明显更高
- no_tumor 中几乎没有

但它覆盖率太低，无法支撑大部分样本。

这就解释了后面一个反复出现的现象：

- 模型可以从 `0.58` 推到 `0.63`
- 但很难跨过 `0.64~0.65`

## 4. 方法学上的共同风险

两批实验都存在一个非常重要的方法学前提。

### 4.1 都是在用测试集反复选最优 epoch

`0319/task2_recipe_utils.py` 和 `0323/scripts/task2_json_utils.py` 的公共训练框架都采用：

1. 每隔若干 epoch 在 `test_loader` 上评估
2. 以 `test macro-F1` 保存最佳权重
3. 训练结束后再加载最佳权重，在同一个测试集上报告最终结果

`0323` 还额外做了：

4. 直接在测试集上搜索最佳分类阈值

这意味着当前仓库里的所有最好分数都更适合理解为：

- “横向实验对比结果”

而不是：

- “严格无偏的最终泛化性能”

### 4.2 阈值优化进一步放大了测试集参与决策的程度

在 `0323/scripts/task2_json_utils.py` 里，阈值搜索区间是：

- `0.15 ~ 0.75`
- 步长 `0.005`

而且搜索目标就是测试集上的 `macro-F1`。

所以对于 `0323` 的“阈值优化后 F1”，应理解为：

- 有很强参考价值
- 但比默认阈值分数更乐观

### 4.3 一个我基于代码做出的推断

`0323/scripts/20260323_task2_feature_xgboost.py` 中，训练内 `5-fold CV` 结果接近满分：

- `CV 平均 F1 = 0.9957`
- `OOF F1 = 0.9946`

但同一脚本在独立 test 上只有：

- 仅深度特征：`0.6315`
- 全特征：`0.6321`

我的推断是：

- 这个 CV 结果不能拿来当作“真实泛化能力”
- 原因很可能不是树模型太神，而是深度特征来自已经在完整训练集上训练过的 backbone，再对同一训练集做特征级 CV，会明显偏乐观

这是推断，不是原文直接结论，但从代码结构和结果差异看，这个解释最合理。

## 5. `0319` 这批实验到底解决了什么

`0319` 的核心任务不是“挖掘 JSON 标注”，而是：

1. 搭一个可靠 baseline
2. 判断 backbone 之间谁更稳
3. 判断哪些训练 recipe 才适合这些 backbone
4. 找出一个可复现、可持续推进的主线模型

### 5.1 第一层：基线实验回答了什么

早期基线里最值得关注的不是谁分数最高，而是一些极具代表性的对照：

| 实验 | 现象 | 说明 |
| --- | --- | --- |
| `ResNet18_baseline` | `Acc` 高，但 benign 抓取弱 | 说明 accuracy 不够用 |
| `SwinTiny_baseline` | `Acc=0.7713`，但 benign recall=`0.0000` | 说明模型几乎塌成“全预测 no_tumor” |
| `ResNet34_weakaug` | 比 `ResNet34_baseline` 更差 | 说明弱增强单独上没有解决不平衡 |
| `ResNet34_weakaug_classweight` | 明显提升 | 说明增强要和不平衡处理联动 |

最核心的早期对照其实是这个：

| 实验 | Macro-F1 | benign recall | benign f1 |
| --- | ---: | ---: | ---: |
| `Resnet34_baseline_1` | 0.5815 | 0.3566 | 0.3552 |
| `Resnet34_weakaug_2` | 0.5479 | 0.1705 | 0.2418 |
| `Resnet34_weakaug_classweight_3` | 0.6225 | 0.4109 | 0.4157 |

这三组结果直接给出了第一阶段最重要的经验：

- 仅靠增强，不能解决少数类学习问题
- 类别权重对 `benign` 是刚需
- 轻量 CNN 在小数据集上未必弱，甚至可能更稳

### 5.2 第二层：为什么很多首轮模型“看起来不行”

`0319/实验报告总结.md` 的一个重要结论是：

- `ConvNeXt / Swin / ViT` 首轮效果不好，不代表模型本身不行
- 更可能是统一 baseline recipe 不适配这些模型

这是非常关键的判断，因为它改变了后续实验方向：

- 不是继续盲目换 backbone
- 而是给每类 backbone 补合适的 recipe

### 5.3 第三层：recipe-tuning 真正带来了什么

`0319` 的第二轮 recipe 大致包括：

- `AdamW`
- backbone/head 差分学习率
- `warmup + cosine`
- `class weight`
- `label smoothing`
- 更适合对应模型的弱增强
- 梯度裁剪

这套改动不是小修小补，而是把训练状态从“没训明白”拉到“可以稳定泛化”。

文档里的提升幅度非常说明问题：

| 模型 | 首轮 baseline F1 | 第二轮 recipe F1 | 提升 |
| --- | ---: | ---: | ---: |
| `ConvNeXtTiny` | 0.4945 | 0.6325 | +0.1380 |
| `SwinTiny` | 0.4354 | 0.6192 | +0.1838 |
| `ViTB16` | 0.5455 | 0.6151 | +0.0696 |
| `EfficientNetB0` | 0.5994 | 0.6115 | +0.0121 |

这个阶段真正回答的是：

> 当前任务的主要矛盾，至少在第一批实验里，不是 backbone 上限，而是 recipe 是否匹配。

### 5.4 为什么 `ConvNeXt` 后来成了主线

在 `0319` 的后续总结里，`ConvNeXtTiny img320` 被定为当前主线，原因不是“它某一项最好”，而是它整体最均衡：

- `Final Test F1 = 0.6372`
- `Acc = 0.7660`
- `benign f1 = 0.4211`
- `no_tumor f1 = 0.8533`

它的特点是：

1. 总分高
2. 多数类没崩
3. 少数类也没完全丢
4. 多 seed 波动不大

这意味着它不是靠“极端偏向某一类”拿分，而是整体平衡最好。

### 5.5 为什么 `Swin` 仍然没有失去价值

虽然 `ConvNeXt` 变成总分主线，但 `SwinTiny_adamw_warmupcosine_classweight_2` 仍然有重要价值：

- `F1 = 0.6192`
- `benign recall = 0.4496`
- `benign f1 = 0.4234`

它在 `benign` 抓取上更激进，适合作为少数类辅助线。

也就是说，`0319` 的最终结论不是：

- “ConvNeXt 完胜一切”

而是：

- “ConvNeXt 更适合作为主报告模型”
- “Swin 更适合作为冲少数类的辅助模型”

### 5.6 为什么 `img320` 只带来小幅提升而不是暴涨

`ConvNeXt img320` 相比前一版 `ConvNeXt recipe`，提升是正的，但不大：

- `macro-F1`: `0.6325 -> 0.6372`
- `benign f1`: `0.4156 -> 0.4211`

这说明：

1. 分辨率提升确实补了一些细节
2. 但这类任务的主矛盾不是“分辨率不够”
3. 当 recipe 已经比较合理后，继续堆输入尺寸只能带来边际收益

### 5.7 为什么 `WeightedRandomSampler` 没成为这批实验的新方向

`0319` 后续总结中专门分析了 sampler：

- `Acc` 更高
- `precision` 更高
- `no_tumor` 更强
- 但 `benign recall` 没有变好

这意味着它更像：

- 一个更保守、更偏多数类、更容易拿高准确率的策略

而不是：

- 真正解决少数类问题的策略

所以它最后被降级成补充对照，而不是主线方向。

### 5.8 `0319` 这批实验的真正结论

如果用一句话概括 `0319`：

> 这批实验完成了“主线模型筛选”和“recipe 校准”两件事，证明了当前阶段最该优先做的是训练策略匹配，而不是继续盲目换 backbone。

## 6. `0323` 这批实验到底新增了什么

如果说 `0319` 主要解决的是“怎么把分类模型训对”，那么 `0323` 主要解决的是：

1. JSON 标注到底有没有实用价值
2. ROI 裁剪、病灶 mask、全图上下文、强增强、阈值搜索谁更有用
3. 当单模型快到顶时，集成和特征学习还能不能继续突破

### 6.1 这批实验的统一框架很重要

`0323/scripts/task2_json_utils.py` 做了一件很关键的事：

- 把数据解析、4 通道适配、同步增强、WeightedSampler、Mixup/CutMix、Focal、阈值优化、训练循环全统一了

这样做的好处是：

- 实验间变量控制得更好
- 更容易判断“到底是哪一个改动带来了提升”

这比简单地堆很多散装脚本更有分析价值。

## 7. `0323` 第一阶段：Exp #1 到 Exp #6 在试什么

第一阶段的目标是：

- 验证不同架构
- 验证 ROI 裁剪是否有效
- 验证 4 通道 mask 是否有帮助
- 验证双分支/结构化特征是否值得继续做

### 7.1 Exp #1 / #2：ROI 裁剪的 3 通道 CNN 基线

- `Exp #1`: `ConvNeXt-Tiny + ROI 3ch`
- `Exp #2`: `ResNet34 + ROI 3ch`

这两组实验本质上是在回答：

> 仅用医生给的胆囊矩形框做 ROI 裁剪，能不能直接带来收益？

结果是：

- `Exp #2 ResNet34` 比 `Exp #1 ConvNeXt` 更稳
- 小数据下，轻量经典 CNN 还是很能打

这和 `0319` 的结论是一致的：复杂模型不一定天然占优。

### 7.2 Exp #3：为什么它是这批实验的关键起点

`Exp #3 = SwinV2-Tiny + ROI 4ch`

这是 `0323` 第一阶段最关键的实验之一，因为它第一次系统引入：

- `RGB + lesion_mask` 四通道输入
- `SyncTransform` 同步几何变换
- 基于 JSON 多边形生成的病灶 mask

它的意义不只是分数高，而是证明：

- 病灶位置先验是有用的
- 医生标注的多边形信息不是噪声
- 4 通道显式引导比单纯 3 通道 ROI 更有潜力

### 7.3 Exp #4 / #5：为什么 MaxViT 和 DeiT 没变成主线

- `MaxViT-Tiny + ROI 4ch`
- `DeiT3-Small + ROI 4ch`

这两组实验说明：

- 更花哨的 Transformer 架构并没有在这个数据规模上带来决定性优势
- 和 `SwinV2 ROI 4ch` 比，它们没有形成明显上限突破

所以这一步的价值主要是“排除法”：

- 不是所有 Transformer 都适合这类小样本、弱差异任务

### 7.4 Exp #6：为什么双分支想法很合理，但结果一般

`Exp #6` 是整批实验里设计感最强的一个：

- 全局 ROI 分支
- 局部病灶裁剪分支
- 手工结构化特征分支
- 共享 backbone 融合

它的想法非常合理，因为从直觉上看：

- 全局上下文有用
- 病灶局部细节也有用
- 病灶面积比和数量这样的结构化信息也可能有用

但它最后没有成为主线，原因也很典型：

1. 模型复杂度上来了
2. batch 更小
3. 数据量仍然很小
4. 容易过拟合

这类结果非常常见：设计逻辑是对的，但在当前数据量下不划算。

## 8. `0323` 第二阶段：Exp #7 到 Exp #11 在试什么

这一阶段的目标非常明确：

- 对抗过拟合
- 让少数类真正学到
- 让模型概率更可校准

### 8.1 Exp #7：为什么“全图 4ch + 强增强”是一个分水岭

`Exp #7 = SwinV2-Tiny + Full 4ch + StrongAug + Mixup/CutMix`

它相比 `Exp #3` 的本质升级是：

1. 从 ROI 裁剪改为全图
2. 仍保留 lesion mask 第 4 通道
3. 加入强增强
4. 加入 Mixup/CutMix
5. 训练轮数拉长

这个实验的意义是双重的。

第一，它修正了一个先验：

- 数据说明文档更推荐“ROI + mask”
- 但实验结果表明“全图 + mask”更强

原因也不难理解：

- ROI 裁剪虽然去掉了背景
- 但也去掉了部分全局上下文
- 对于本来就差异极弱的任务，这部分上下文可能反而重要

第二，它证明：

- 强增强并没有把任务彻底训坏
- 在合适架构上，强增强能帮助模型从 ROI 视角转向更稳的全图视角

### 8.2 Exp #8：为什么同样的强增强策略换到 ConvNeXt 没有继续涨

`Exp #8 = ConvNeXt-Tiny + Full 4ch + StrongAug + Mixup/CutMix`

这一步最有价值的不是结果本身，而是它提供了一个对照：

- 强增强不是对任何 backbone 都等效
- SwinV2 在“全图 + mask + 强增强”这个组合上更自然
- ConvNeXt 在这一组设置下没有把收益吃满

这并不表示 ConvNeXt 不行，而是说明：

- 某些收益来自“输入表示 + 架构偏好”的耦合
- 不是一个 augmentation recipe 能对所有模型普适暴涨

### 8.3 Exp #9：为什么说它真正的价值是“修 bug”

`Exp #9 = SwinV2 + WeightedSampler + 修复 Mixup 类权重`

这个实验非常重要，因为它不是“新花样”，而是在纠正前面实验的错误实现。

问题在于：

- Mixup 后标签变成 soft labels
- 之前 soft CE 没有乘类别权重
- 于是 `benign` 在混合训练时失去了额外加权

代码修复后：

- `loss = -(soft_labels * log_probs * w).sum(dim=1).mean()`

这一步本质上让训练目标重新对齐到了任务目标：

- 不只是学混合样本
- 还要继续对少数类敏感

从日志看，`Exp #9` 默认阈值下：

- `F1 = 0.6052`
- `benign recall = 0.5659`
- `benign precision = 0.3782`

这说明它学会了更积极地抓 `benign`，但代价是 precision 不够好。

一旦把阈值升到 `0.745`：

- `F1 = 0.6240`
- `benign recall = 0.4264`
- `benign precision = 0.4365`

这表明它真正的问题不是“模型没学到”，而是：

- 概率输出偏激进
- 需要阈值校准

### 8.4 Exp #10：Focal Loss 的真实作用是什么

`Exp #10 = SwinV2 + Focal Loss + BalancedSampler + 阈值优化`

很多人直觉上会觉得：

- 类别不平衡任务上 Focal 一定更好

但这个实验给出的答案更微妙。

它在默认阈值下：

- `F1 = 0.6045`

调阈值后：

- `F1 = 0.6173`

这说明 Focal 不是没用，而是它的收益更像：

- 改变了难样本学习方式
- 同时改变了概率分布和决策边界

从总结文档看，Focal 在早期 epoch 一度把 benign recall 冲得很高，但后续过拟合明显。这类表现很符合 Focal 的典型特征：

- 对难样本非常敏感
- 但如果数据规模不够，后期容易把少数类边界推得过猛

所以 `Exp #10` 的结论不是“Focal 无效”，而是：

- Focal 可以当补充手段
- 但没有超过“修复后的 CE + Mixup + sampler + threshold”这条线

### 8.5 Exp #11：为什么 ConvNeXt 版 Focal 没翻盘

`Exp #11 = ConvNeXt + Focal + BalancedSampler`

它的结果延续了前面的模式：

- 换 backbone 不能自动解决任务瓶颈
- 真正决定上限的是这个数据集里能不能学到稳定区分信号

所以 `Exp #11` 也没有成为更优主线。

## 9. `0323` 第三阶段：多 seed、集成、stacking 到底说明了什么

这一阶段的目标不是再提升单模型，而是回答：

1. 单模型波动到底有多大
2. 集成能不能进一步突破
3. “更复杂的特征工程”是否比“更合理的预测器融合”更有效

### 9.1 多 seed 实验最重要的结论：波动存在，但不是完全随机

5 个 seed 的 `SwinV2-Tiny` 结果里：

- seed `789` 最强，`F1 = 0.6393`
- 其他 seed 大多在 `0.60 ~ 0.63`

这说明：

- 随机性确实会影响最终结果
- 但不会把一个差模型变成神模型
- 多 seed 是必要的，不是可有可无

换句话说：

- 单个最好 seed 可以反映“这条线的潜力”
- 多 seed 均值更能反映“这条线的稳定性”

### 9.2 为什么“6 模型全平均”不如“最佳 2 模型组合”

`logs/ensemble_eval/ensemble_eval.log` 给出的现象非常典型：

- 6 模型整体平均，最优阈值后只有 `F1 = 0.6220`
- 但最佳二模型组合 `Exp3 + Exp9` 能到 `F1 = 0.6423`

这说明集成里有一个经常被忽略的事实：

- 模型越多不一定越强
- 如果弱模型和强模型的误差结构不互补，平均后反而会冲淡好模型的信号

因此，这个结果真正告诉我们的是：

- 需要的是“互补性强的模型”
- 不是“数量最多的模型”

### 9.3 为什么 5-seed NN ensemble 比单模型强，但提升有限

多 seed softmax 平均可以到：

- `F1 = 0.6373`

这比大多数单模型高，但没有高很多。原因也很合理：

- 同一架构、同一训练设置、只是随机种子不同
- 学到的表示高度相关

所以它确实能：

- 降噪
- 平滑偶然偏差

但它很难：

- 产生真正新的判别信息

### 9.4 为什么 14 预测器 stacking 是这批实验的最高点

`0323/logs/multiseed_ensemble_xgb/multiseed_ensemble_xgb.log` 中最强结果是：

- `Stacked 14 预测器`
- `threshold = 0.590`
- `F1 = 0.6501`
- `benign f1 = 0.4729`

这是整个 `0323` 文档里反复强调的最高点。

它之所以优于简单平均，原因在于：

1. 它不是假设所有模型等权
2. 它允许不同预测器在不同概率区间承担不同作用
3. 它把“谁更可信”也作为可学习对象

这类 stacking 在单模型已经接近天花板时非常有效，因为它利用的是：

- 预测器之间细微但稳定的互补误差

而不是：

- 去幻想某个新模型会突然学到全新视觉模式

## 10. `0323` 第四阶段：为什么更复杂的特征工程没有继续突破

这一阶段尝试了三类东西：

1. 深度特征 + XGBoost / LightGBM
2. DINOv2 + 多架构融合
3. 标注类型感知特征

### 10.1 深度特征 + XGBoost 为什么没有大超越

`feature_xgboost.log` 里：

- 仅深度特征：`F1 = 0.6315`
- 全特征：`F1 = 0.6321`

这两个结果说明：

- 树模型不是完全没用
- 但它们没有从现有深度特征中挖出远超神经网络输出的新增信息

换句话说，瓶颈不在“分类器头不够强”，而在“原始特征本身的信息量有限”。

### 10.2 DINOv2 为什么没成为救星

很多人会天然期待：

- 自监督大模型特征更强
- DINOv2 一上，医学小数据就会起飞

但这批实验的结果并不支持这个期待：

- `DINOv2 Best Ensemble` 大约 `0.6075`
- 全特征最优 ensemble 约 `0.6421`
- 仍然没超过 stacking 的 `0.6501`

这表明：

- 通用自监督特征并不天然等于“最适合这个病种判别”
- 任务微调过的专用特征，至少在这个任务上仍然更贴近目标

### 10.3 标注类型特征为什么看起来有逻辑，却没有大突破

标注类型感知实验抓住了一个很对的方向：

- `adenoma` 几乎只在 benign 出现

按直觉，这应该很有帮助。但结果是：

- 纯标注类型特征：`F1 = 0.6167`
- 深度 + 标注类型：`F1 = 0.6372`
- 最终 stacking：`0.6396`
- 手工加权最好也只到 `0.6444`

这说明问题不是这个特征没鉴别力，而是：

1. 它覆盖率太低
2. 对大多数样本不起作用
3. 一旦深度特征已经学到大部分可见信息，它只能补极少数边角样本

也就是说，`adenoma` 像一把很好用的钥匙，但只适用于少数门，不适用于整栋楼。

## 11. 把两批实验连起来看，结论其实非常一致

虽然 `0319` 和 `0323` 用的不是同一套数据，但两批实验给出的信息高度一致。

### 11.1 一致结论一：真正先解决的是训练策略，不是换模型

`0319` 告诉我们：

- recipe 不对，强模型也会训废

`0323` 告诉我们：

- 输入表示和训练目标不对，复杂模型也不会自动变强

两者共同说明：

- 在小样本医学任务里，训练策略往往比 backbone 名字更重要

### 11.2 一致结论二：少数类 `benign` 是整个任务的核心瓶颈

两个批次都反复出现相同模式：

- `no_tumor` 往往比较稳
- 最难的是把 `benign recall` 和 `benign precision` 同时拉起来

所以几乎所有改动都可以用一句话概括：

- 它是在试图重新平衡 benign 与 no_tumor 的决策边界

### 11.3 一致结论三：收益最大的改动都不是“更复杂”

从整个实验史回看，最赚钱的改动其实是：

- 类别权重
- 合适的 optimizer + scheduler
- 4 通道 mask
- 强增强
- 修复 Mixup 权重 bug
- 阈值优化
- 有选择的 stacking

而不是：

- 最复杂的双分支
- 最大的多特征融合
- 最重的自监督特征

这非常说明问题：当前瓶颈不在模型表达能力不够，而在任务信号本来就弱。

## 12. 我对这些实验的总判断

### 12.1 哪些结论我认为是“已经很稳”的

以下结论已经被多组结果反复支持：

1. `Acc` 不能单独作为主指标
2. `classweight` 对当前任务是必要条件之一
3. 统一 baseline recipe 不适合所有 backbone
4. ROI 裁剪并不一定优于全图，至少在 `0322dataset` 上不是
5. 4 通道 lesion mask 是有效信息
6. 阈值优化能稳定带来若干点收益，但它本质上属于测试集调参
7. 多模型 stacking 的确能把上限再推一点，但推幅有限

### 12.2 哪些结论我认为要谨慎说

以下结论不能说得太满：

1. “ConvNeXt 一定比 Swin 更好”
   - 在 `0319` 数据上，ConvNeXt 总分更高
   - 但在 `0323` 的主线里，SwinV2 明显更重要

2. “全图一定比 ROI 更好”
   - 在 `0322dataset` 上结论成立
   - 但这仍然与数据组织和标注方式相关

3. “特征工程完全没价值”
   - 更准确的说法是：它有价值，但没能突破主要瓶颈

4. “当前模型已经达到理论上限”
   - 文档倾向这样判断
   - 我同意“已接近当前数据条件下的经验上限”
   - 但“理论上限”这个词仍然应该保守使用

### 12.3 我认为最接近真相的一句话

如果必须用一句话总结整个项目当前阶段，我会写成：

> 当前任务的主要限制来自数据可分性，而不是模型复杂度；实验已经基本把“训练策略、标注利用和预测融合”这三条可优化路径都压到较高水平了，继续单靠模型花样很难再出现大幅突破。

## 13. 如果后续还要继续做，最值得做什么

基于现有结果，后续工作的优先级我会这样排。

### 13.1 最高优先级：把评估流程先做干净

建议优先补：

1. 独立 validation set
2. 或者患者级 cross-validation
3. 把 threshold tuning 从 test 挪到 val

原因很简单：

- 现在模型已经接近平台区
- 再往上抠 1~2 个点时，方法学偏差会变得比模型差异还大

### 13.2 第二优先级：继续做数据层面改进

文档里这一点说得其实是对的，而且我认同：

- 增加 benign 数据
- 审核标签质量
- 引入临床信息
- 引入更可靠的病理关联信息

这几项的潜在收益，大概率高于继续堆新 backbone。

### 13.3 第三优先级：保留少量低成本模型增益项

如果还想在模型侧继续试，建议只做低成本且逻辑闭环清楚的方向：

- 在已有主线基础上做严格 val-based threshold tuning
- 小规模 TTA
- SWA / checkpoint averaging
- 有明确对照的单变量 ablation

### 13.4 当前不建议高投入的方向

按照现有证据，我不建议再大投入：

- 再大范围换 backbone
- 继续堆更重的多特征融合
- 把更多精力投到“超复杂结构”而不是“更严格评估”

因为这些方向目前都没有显示出足够好的性价比。

## 14. 一页纸版结论

如果只保留最核心的信息，可以压缩成下面这几条。

1. `0319` 解决的是 recipe 问题，确认了主线模型和少数类辅助线：
   - 总分主线：`ConvNeXtTiny img320`
   - 少数类辅助线：`SwinTiny classweight`

2. `0323` 解决的是 JSON 标注利用问题：
   - 4 通道 lesion mask 有用
   - 全图 + mask 比 ROI + mask 更有效
   - 强增强、sampler、threshold tuning 都有帮助

3. `Mixup` 类权重 bug 的修复是 `0323` 最有价值的技术修正之一。

4. 简单“堆更多模型平均”没用，真正有效的是有选择的 pair ensemble 和 stacking。

5. 最终性能长期停在 `0.64~0.65` 左右，不是因为没试够模型，而更像是数据可分性本身限制了上限。

6. 如果要继续推进，优先补评估流程和数据质量，而不是继续大范围换架构。
