# Task2 实验报告总结

## 1. 实验目标

本轮实验聚焦 `Task2: benign(良性肿瘤) vs no_tumor(非肿瘤性息肉)` 二分类任务，目标是：

- 在统一数据集和统一评估方式下，对比不同 backbone、loss 和 augmentation 的效果
- 重点观察 `macro precision / macro recall / macro F1`，尤其关注少数类 `benign` 的识别能力
- 分析各模型的收敛、波动、过拟合和失效原因
- 明确后续最值得继续推进的主线方案

## 2. 数据与统一实验设置

### 2.1 数据划分

- 训练集：1303 张
- 测试集：564 张
- 类别分布：
  - train: `benign=309`, `no_tumor=994`
  - test: `benign=129`, `no_tumor=435`

该任务存在明显类别不平衡，约为 `1:3`。

此外，本轮补充做了一组 `1:1` 平衡数据实验，但它不是重新对全体样本做随机 `7/3` 划分，而是：

- 在原始 train 内，将 `no_tumor` 下采样到与 `benign` 相同数量
- 在原始 test 内，也将 `no_tumor` 下采样到与 `benign` 相同数量

对应数据为：

- 平衡训练集：`618` 张，`benign=309`, `no_tumor=309`
- 平衡测试集：`258` 张，`benign=129`, `no_tumor=129`

因此，这组 `1:1` 实验可以用来看“平衡分布下模型表现如何”，但它和原始分布测试集上的分数不能直接横向视为同一排行榜。

### 2.2 首轮统一 baseline 设置

首轮 baseline 实验默认采用如下统一设置：

- 数据来源：`dataset/Processed/task_2_train.xlsx` 与 `task_2_test.xlsx`
- 输入尺寸：统一 resize 到 `224 x 224`
- 预训练：全部使用 ImageNet 预训练权重
- 训练方式：默认全量微调
- 优化器：`Adam`
- 学习率：`1e-3`
- `weight_decay`: `1e-4`
- `batch_size`: `32`
- `num_epochs`: `100`
- 每 `3 epoch` 在测试集上评估一次
- 以 `test macro-F1` 保存最佳权重

### 2.3 第二轮模型专用 recipe 设置

针对首轮中 `ConvNeXt / Swin / ViT / EfficientNet` 暴露出的 recipe 不匹配问题，第二轮补充了模型专用训练配方，核心包括：

- `AdamW`
- backbone / head 差分学习率
- `warmup + cosine` 学习率调度
- `class weight`
- `label smoothing`
- 更贴近模型特性的弱增强
- 更短训练轮数与梯度裁剪

因此，本报告后半段的“第二轮 recipe-tuned 实验”不再和首轮 baseline 共享完全相同的训练配置。

### 2.4 评估指标

统一记录：

- `Acc`
- `Precision(macro)`
- `Recall(macro)`
- `F1(macro)`
- 分类报告：`benign / no_tumor` 的 precision、recall、f1-score

### 2.5 方法学注意事项

当前实验只有 `train / test`，没有独立 `val`。因此：

- 测试集被反复用于选最佳 epoch
- 当前结果适合用于横向比较
- 但不应视为严格意义上的最终泛化性能

换句话说，当前分数有参考价值，但整体会略偏乐观。

## 4. 关键结论

### 4.1 当前最佳方案

当前最佳实验为：

- `20260319_task2_ConvNeXtTiny_adamw_warmupcosine_classweight_2`

其优势不只是总分最高，更重要的是：

- `macro-F1` 最高：`0.6325`
- `Accuracy` 也保持较高：`0.7606`
- `benign f1` 处于第一梯队：`0.4156`
- `no_tumor f1` 很高：`0.8495`

这说明它不是靠单纯偏向多数类拿分，而是实现了相对均衡的识别。

这里的“当前最佳”特指：

- 在原始数据分布下训练
- 在原始测试集分布下评估

如果只看平衡测试集，`balance1to1_4` 的 `macro-F1 = 0.6535` 会更高；但它回到原始测试集后只有 `0.5223`，因此不能替代当前主线结论。

### 4.2 单纯看 Accuracy 会误导

例如：

- `Resnet18_baseline_1` 的 `Acc = 0.7713`
- `SwinTiny_baseline_1` 的 `Acc = 0.7713`

但这两个实验并不等价：

- `Resnet18_baseline_1` 还能识别一部分 benign
- `SwinTiny_baseline_1` 的 `benign recall = 0.0000`

也就是说，`SwinTiny` 几乎退化成“全部预测 no_tumor”，高 `Acc` 只是因为测试集多数类占比高。

因此，本任务更应该优先关注：

- `macro-F1`
- `benign recall`
- `benign f1`

而不是单独看 `Acc`。

### 4.3 ResNet 系列已经跑出主线方向

ResNet 系列结果说明了三件事：

- `ResNet34` 比 `ResNet18` 更平衡
- `classweight` 对 benign 识别有明显帮助
- `weakaug` 不能单独上，必须和不平衡处理联动

其中最关键的比较是：

| 实验 | Macro-F1 | benign recall | benign f1 |
| --- | ---: | ---: | ---: |
| `Resnet34_baseline_1` | 0.5815 | 0.3566 | 0.3552 |
| `Resnet34_weakaug_2` | 0.5479 | 0.1705 | 0.2418 |
| `Resnet34_weakaug_classweight_3` | 0.6225 | 0.4109 | 0.4157 |

这个对比非常清晰：

- 弱增强单独使用：变差
- 弱增强 + 类别权重：显著变好

### 4.4 第二轮模型专用 recipe 的核心结论

第二轮专用 recipe 基本把首轮的几个“问题模型”都拉回到了可用区间，而且提升幅度非常明确：

| 模型 | 首轮 baseline Macro-F1 | 第二轮 recipe Macro-F1 | 提升 |
| --- | ---: | ---: | ---: |
| `ConvNeXtTiny` | 0.4945 | 0.6325 | +0.1380 |
| `SwinTiny` | 0.4354 | 0.6192 | +0.1838 |
| `ViTB16` | 0.5455 | 0.6151 | +0.0696 |
| `EfficientNetB0` | 0.5994 | 0.6115 | +0.0121 |

这组结果说明：

- 首轮里 `ConvNeXt / Swin / ViT` 的差，不完全是模型本身不行
- 更关键的问题是统一 baseline recipe 与这些模型不匹配
- 一旦换成更合理的 `AdamW + diff lr + warmup cosine + class weight + label smoothing + weakaug`，模型上限会明显变化

其中两个最关键的观察是：

- `ConvNeXtTiny` 直接成为当前原始分布测试集上的总分第一
- `SwinTiny` 从“几乎完全塌成多数类预测”恢复到当前 `benign recall / benign f1` 最强

### 4.5 `1:1` 平衡 train/test 实验的结论

这次补充实验有两个表面上矛盾、实际上很一致的现象：

- 在平衡测试集上，`macro-F1 = 0.6535`，高于当前原始分布主线结果 `0.6225`
- 但把这次训练得到的最佳权重放回原始测试集，`macro-F1` 掉到 `0.5223`

这说明：

- 把 test 也改成 `1:1`，会显著改变评估难度和指标形态
- 平衡测试集更容易鼓励模型追求两类“平均命中”
- 但这不代表它对原始真实分布更有泛化能力

更具体地说，这个模型在原始测试集上表现为：

- `benign recall` 大幅提高到 `0.7209`
- 但 `no_tumor recall` 明显下降到 `0.4920`
- 最终整体 `macro-F1` 和 `Acc` 都低于原主线

因此，这个实验更像是在说明“强平衡训练会把决策边界明显推向 benign”，而不是说明“1:1 train/test 更优”。

## 5. 波动与损失下降情况分析

### 5.1 训练损失下降情况

当前实验大致分成两类：

#### 第一类：训练损失很快降到很低，训练精度几乎记满

代表实验：

- `Resnet18_baseline_1`: `loss=0.0003`, `train acc=1.0000`
- `Resnet34_baseline_1`: `loss=0.0163`, `train acc=0.9923`
- `EfficientnetB0_baseline_1`: `loss=0.0148`, `train acc=0.9962`
- `Densenet121_baseline_1`: `loss=0.0324`, `train acc=0.9885`

这类实验说明：

- 模型已经把训练集记得很充分
- 训练误差几乎被清空
- 但测试 `macro-F1` 没有同步提升

这就是典型的过拟合轨迹。

补充一个对照：

- `Resnet34_weakaug_classweight_balance1to1_4`: `loss=0.1790`, `train acc=0.9207`

它没有像最激进的 baseline 那样把训练集几乎记满，但由于训练样本数被压缩到 `618`，最佳点反而提前到了 `epoch33`，说明“小数据 + 全量微调”仍然会很快进入过拟合区间。

#### 第二类：训练损失没有明显降到很低，训练精度也不高

代表实验：

- `Resnet34_weakaug_2`: `loss=0.5054`, `train acc=0.7698`
- `ConvNeXtTiny_baseline_1`: `loss=0.4328`, `train acc=0.7866`
- `SwinTiny_baseline_1`: `loss=0.5455`, `train acc=0.7629`
- `ViTB16_baseline_1`: `loss=0.4974`, `train acc=0.7974`

这类实验说明：

- 不是“训练集记满了”
- 更像是优化没收敛、recipe 不合适，或者增强/模型与当前数据规模不匹配

也就是说，不能把所有表现差的实验都简单归因为过拟合。

#### 第三类：专用 recipe 后进入“可训练、可泛化”的中间状态

代表实验：

- `ConvNeXtTiny_adamw_warmupcosine_classweight_2`: `loss=0.2822`, `train acc=0.9985`
- `SwinTiny_adamw_warmupcosine_classweight_2`: `loss=0.2944`, `train acc=0.9946`
- `ViTB16_adamw_warmupcosine_classweight_2`: `loss=0.2761`, `train acc=0.9992`
- `EfficientnetB0_weakaug_classweight_warmupcosine_2`: `loss=0.2216`, `train acc=0.9816`

这类实验说明：

- 模型最终仍然能把训练集学得很充分
- 但相比首轮统一 recipe，它们的泛化并没有同步崩掉
- 调整优化器、学习率节奏和不平衡处理后，模型的有效性能被释放出来了

### 5.2 指标波动情况

波动大的根本原因主要有三点：

- `benign` 在测试集只有 `129` 张，少数类太少
- 当前使用的是 `macro-F1`，它对少数类波动非常敏感
- 每 `3 epoch` 就在 `test` 上评估，测试噪声会被频繁看到

这意味着：

- benign 多识别对几张，`macro-F1` 就会明显上升
- benign 少识别对几张，`macro-F1` 就会明显回落

因此很多实验呈现出：

- `Acc` 看上去稳定
- `macro-F1`、`benign recall` 上下抖动明显

这并不是指标“坏了”，而是少数类样本量太小，导致宏平均指标天然更敏感。

### 5.3 过拟合现象

当前多数 CNN 都存在明显过拟合，表现为：

- 最优 epoch 普遍远早于 `epoch100`
- 到训练后期，`train loss` 持续下降
- 但 `test macro-F1` 反而下滑

典型例子：

- `Resnet18_baseline_1`：最优 `epoch12`，`overfit gap=0.0726`
- `Resnet34_baseline_1`：最优 `epoch27`，`overfit gap=0.0708`
- `EfficientnetB0_baseline_1`：最优 `epoch12`，`overfit gap=0.0845`
- `Resnet34_weakaug_classweight_balance1to1_4`：最优 `epoch33`，平衡 test 上 `overfit gap=0.0569`
- `ConvNeXtTiny_adamw_warmupcosine_classweight_2`：最优 `epoch36`，`overfit gap=0.0046`
- `SwinTiny_adamw_warmupcosine_classweight_2`：最优 `epoch39`，`overfit gap=0.0283`
- `ViTB16_adamw_warmupcosine_classweight_2`：最优 `epoch21`，`overfit gap=0.0288`
- `EfficientnetB0_weakaug_classweight_warmupcosine_2`：最优 `epoch46`，`overfit gap=0.0213`

这说明：

- 100 epoch 对当前数据规模偏长
- 固定学习率、无 scheduler、无 early stop 的策略容易后期过拟合
- 当训练集进一步缩小到 `618` 张时，最佳 epoch 会更早到来
- 专用 recipe 可以明显减缓“训练越来越好、测试越来越差”的失控趋势

但是也要注意：

- `Resnet34_weakaug_classweight_3` 虽然也有过拟合，`gap=0.0627`
- 但相比纯 baseline，它后期仍保留了更好的泛化能力

原因是：

- 弱增强提高了训练难度
- 类别权重抑制了多数类主导
- 模型没有那么快把训练集“背死”

## 6. 为什么有些模型效果好，有些效果不好

### 6.1 为什么首轮 `ResNet34 + weakaug + classweight` 效果最好

这个组合之所以最好，是因为它同时解决了两个核心问题：

#### 问题一：少数类 benign 太难学

`classweight` 让损失对 benign 更敏感，直接推动模型提高 benign 召回。

#### 问题二：基础 CNN 容易过快记住训练集

`weakaug` 提高了训练样本的多样性，减缓了简单记忆。

两者叠加后：

- 模型既不会完全倒向 `no_tumor`
- 又不会像纯 baseline 一样太快记训练集

这是首轮统一 baseline 中最合理的平衡点。

补充一点：

- 该结论成立的前提是“原始训练分布 + 原始测试分布”
- 一旦把 train/test 都强行改成 `1:1`，模型最优行为会转向“更 aggressively 抓 benign”
- 这会抬高平衡测试集上的 `macro-F1`，但不一定提升真实分布下的效果

### 6.2 为什么 `weakaug` 单独上会变差

`weakaug` 只增加了训练难度，但没有处理类别不平衡。

结果就是：

- 模型更难优化
- 但仍然倾向多数类
- 最终 benign 识别能力反而变弱

这说明在当前任务里：

- 数据增强本身不是万能的
- 必须配合不平衡处理一起用

同样地，这也解释了为什么 `balance1to1_4` 没有成为新的主线：

- 它虽然通过下采样把类别频次拉平了
- 但同时也丢掉了大量 `no_tumor` 样本
- 最终模型学到的是“更偏 benign 的边界”，而不是“在真实分布上更稳的边界”

### 6.3 为什么 `DenseNet121` 和 `EfficientNet-B0` 表现不错

这两个模型的共同特点是：

- 对局部纹理和细节建模能力较好
- 在小到中等规模数据上通常比更重的 Transformer 更稳

其中：

- `DenseNet121` 的总体准确率和多数类性能都很强
- `EfficientNet-B0` baseline 已经不错
- `EfficientNet-B0` 在第二轮 recipe 后进一步提升到 `macro-F1 = 0.6115`

它们说明：当前任务仍然是 CNN 更占优。

### 6.4 为什么 `ConvNeXt / Swin / ViT` 首轮不理想，但第二轮变好了

首轮这三类模型整体偏弱，但第二轮结果证明，问题更偏向 recipe 而不是模型本身。

#### `ConvNeXtTiny`

首轮更像是 recipe 没调好：

- 训练精度不高
- 训练损失也不低
- 没有完全收敛

说明当前固定 `Adam + lr=1e-3 + 100 epoch` 对它并不合适。

第二轮换成专用 recipe 后：

- `macro-F1: 0.4945 -> 0.6325`
- 直接成为当前总榜第一

这说明 ConvNeXt 并不是“不适合这个任务”，而是“不能按统一 baseline 粗暴训练”。

#### `SwinTiny`

首轮几乎直接塌成多数类预测：

- `benign recall = 0`
- `macro-F1 = 0.4354`

这意味着它在当前设置下没有真正学到有用的少数类判别边界。

第二轮换 recipe 后：

- `macro-F1: 0.4354 -> 0.6192`
- `benign recall = 0.4496`
- `benign f1 = 0.4234`

这说明 Swin 对优化器、学习率和类别不平衡处理非常敏感。

#### `ViTB16`

首轮主要问题更像是：

- 数据量偏小
- 默认超参不适合
- 全量微调成本高但收益低

第二轮后：

- `macro-F1: 0.5455 -> 0.6151`

它仍然没有超过最强 CNN，但已经从“性价比不高”变成了“可以作为有效对照线”。

## 7. 为什么过拟合会这么严重

当前过拟合严重，原因是多因素叠加：

### 7.1 数据量有限

总训练样本只有 `1303` 张，且 benign 只有 `309` 张。

对于全量微调的深度模型来说，这个规模并不宽裕。

如果进一步做 `1:1` 下采样，训练样本会直接降到 `618` 张，过拟合风险会进一步放大。

### 7.2 类别不平衡明显

`1:3` 的类别比会让模型天然更愿意学习多数类。

如果 loss 或采样不处理这个问题：

- 模型更容易通过“多数类正确”来获得较低训练损失
- 却不一定真正学会 benign

### 7.3 全量微调 + 长训练轮数

当前大部分实验都是：

- 全量微调
- 100 epoch
- 无 scheduler
- 无 early stop

这会让模型在中后期更容易贴紧训练数据。

### 7.4 当前输入尺寸统一为 224

处理后图像原本是 `320 x 320`，当前全部压到 `224 x 224`。

这本身未必错误，但可能带来两种副作用：

- 一部分局部信息被压缩
- 模型更容易依赖粗粒度模式，而不是稳定的病灶特征

### 7.5 测试集被反复使用

虽然这不会制造训练意义上的过拟合，但会制造“选择意义上的过拟合”：

- 反复看 test
- 反复按 test 选最好 epoch

会让结果更容易贴合这份测试集的偶然波动。

## 8. 本轮实验最值得保留的思考

### 8.1 不是模型越新越大越好，但 recipe 比模型名更重要

本轮结果已经非常明确：

- 当前最优仍然是中等规模 CNN
- 但 `Transformer` 并不是天然不行
- 更准确的说法是：`错误 recipe` 会让模型失真，`合适 recipe` 才能看到真实上限

这说明当前瓶颈主要不是“模型不够先进”，而是：

- 数据规模有限
- 类别不平衡
- 训练策略需要更贴近任务本身

### 8.2 不能只看 Accuracy

本任务的关键不是把大多数 `no_tumor` 判对，而是要把 benign 稳定抓出来。

因此后续优化应优先看：

- `macro-F1`
- `benign recall`
- `benign f1`

### 8.3 少数类建模是主矛盾

从 `classweight` 的收益可以看出：

- 不平衡处理是必须做的
- 这不是锦上添花，而是决定任务上限的关键项

### 8.4 augmentation 必须和任务配套

增强不是越强越好，也不是单独上就一定涨。

当前最有价值的经验是：

- `weakaug` 单独使用：不理想
- `weakaug + classweight`：效果很好

这说明 augmentation 的收益取决于它是否和 loss、采样、类别分布协同。

### 8.5 平衡训练可以试，但不应平衡测试集

这次补充实验给出的最重要经验之一是：

- 可以在 train 端做重采样、下采样或加权
- 但 test 最好保持原始分布

原因很简单：

- test 的职责是反映真实部署分布下的表现
- 如果 test 也被改成 `1:1`，指标会更“均衡”、更“好看”
- 但这可能掩盖模型在原始多数类分布下的真实误报/漏报结构

## 9. 后续推进建议

### 9.1 主线方案

当前建议把以下方案作为后续主线：

- `ConvNeXtTiny_adamw_warmupcosine_classweight_2`

理由：

- 当前总分最高
- 原始分布测试集上 `macro-F1 = 0.6325`
- 训练与泛化更稳定，`overfit gap` 很小
- 说明第二轮 recipe 对它非常匹配

同时建议保留两条辅助主线：

- `SwinTiny_adamw_warmupcosine_classweight_2`
  - 当前 `benign recall / benign f1` 最强，适合继续冲少数类
- `Resnet34_weakaug_classweight_3`
  - 作为最稳的经典 CNN 参考线，便于判断新改动是真提升还是 recipe 噪声

### 9.2 下一轮优先做的实验

建议按下面顺序推进：

#### 第一优先级：继续优化当前最优主线 `ConvNeXtTiny`

- `ConvNeXtTiny_adamw_warmupcosine_classweight_img320`
  - 目的：验证更高分辨率是否还能继续抬升 `benign f1`
- `ConvNeXtTiny_adamw_warmupcosine_focalloss`
  - 目的：继续强化少数类和难样本
- `ConvNeXtTiny_adamw_warmupcosine_weightedsampler`
  - 目的：比较 sampler 和 classweight 的稳定性
- `ConvNeXtTiny_adamw_warmupcosine_tta`
  - 目的：低成本抬升最终测试表现

#### 第二优先级：围绕 `SwinTiny` 冲少数类指标

- `SwinTiny_adamw_warmupcosine_focalloss`
- `SwinTiny_adamw_warmupcosine_img224_256aug`
- `SwinTiny_adamw_warmupcosine_weightedsampler`

因为它当前已经证明自己在 `benign recall / benign f1` 上有最高潜力。

#### 第三优先级：保留 `ResNet34` 与 `EfficientNet` 作为稳定参考线

- `Resnet34_weakaug_classweight_img320`
- `Resnet34_weakaug_weightedsampler`
- `EfficientNetB0_weakaug_classweight_img320`
- `EfficientNetB0_weakaug_focalloss`

### 9.3 暂不建议作为主线的方向

当前不建议继续重点投入：

- 旧版 `20260318_task2_SwinTiny_baseline_1`
- 旧版 `20260318_task2_ViTB16_baseline_1`
- 旧版 `20260318_task2_ConvNeXtTiny_baseline_1`

这些“统一 baseline recipe”版本已经完成了历史使命：它们证明了 recipe 不匹配会导致严重低估模型上限。后续不需要继续在这些旧版脚本上重复投入。

### 9.4 评估流程改进建议

即使当前不重做数据文件，也建议后续逐步改成：

- `train` 用于训练
- `val` 用于选 epoch / loss / threshold
- `test` 只做最后一次报告

如果暂时还不想单独建 `val` 文件，也建议至少：

- 在 train 内部做一层交叉验证或临时划分
- 把 test 从“反复调参”的流程里拿出来

## 10. 最终结论

本轮实验已经给出一个比较清晰的答案：

- 当前任务最有效的不是更大的 Transformer，而是更合适的 CNN 配方
- 真正影响上限的关键因素不是 backbone 本身，而是少数类 benign 的学习能力
- 单独的数据增强不能解决问题，必须和不平衡处理联动
- 目前原始分布测试集下最优方案已经更新为 `ConvNeXtTiny + AdamW + diff lr + warmup cosine + class weight + label smoothing + weakaug`
- `SwinTiny` 和 `ViT-B/16` 在专用 recipe 下也被明显救活，说明先前的低分并不能直接说明模型无效
- 把 train/test 都改成 `1:1` 会让平衡测试集分数更高，但不会自动提升原始分布下的泛化效果

从研究推进角度看，下一阶段不应继续大范围盲试模型，而应围绕当前最优主线做针对性改进：

- 保留更高分辨率信息
- 继续优化少数类损失或采样
- 如果尝试平衡数据，应优先只动训练集，不动测试集
- 用 scheduler 和更合理的停止策略抑制过拟合
- 减少测试集参与模型选择

如果后续继续推进得当，当前结果仍有进一步提升空间，尤其是 `benign recall` 和 `benign f1` 还有明显可优化余地。
